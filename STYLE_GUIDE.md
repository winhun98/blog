# Blog Writing Style Guide

## 문체

### 톤: 연구 일지체
- **"~했다", "~이다" 체.** 존댓말 아님. 일기/연구노트처럼 쓴다.
- 논문 투 아니고, AI가 쓴 것 같은 매끄러운 설명조도 아니다.
- **사람 냄새가 나야 한다.** 중간중간 개인 생각, 느낀 점, 의문을 짧게 끼워넣는다.

예시:
> T-learner를 처음 돌렸을 때 ITE 분포가 너무 깔끔하게 나와서 오히려 의심이 들었다. 관측 데이터에서 이렇게 예쁜 결과가 나올 리가 없으니까.

> 솔직히 이 부분은 아직 확신이 없다. 다만 실험적으로는 일관된 결과가 나오고 있어서 일단 이 방향으로 밀어보기로 했다.

### 수학 중심
- 방법론 글은 **수식이 backbone**. 직관 설명은 수식 전에 한 줄, 수식 후에 해석 한 줄.
- 수식 없이 "개념적으로 이해하면~" 식의 얼버무리기 금지.
- 수식 → 코드 → 결과 흐름이 자연스럽게 이어져야 한다.
- LaTeX 인라인 `$...$`, 블록 `$$...$$`.

### 개인 코멘트 삽입 규칙
- 섹션당 1~2회, **짧게**. 한 문장 또는 두 문장.
- 위치: 수식 해석 직후, 실험 결과 직후, 한계점 언급 시.
- 형식: 본문에 자연스럽게 섞는다. 따로 블록이나 인용으로 빼지 않는다.
- 예: "이게 실무에서 은근히 잘 안 잡힌다.", "처음엔 이걸 왜 쓰는지 몰랐다."

## 구조

- **핵심 먼저.** 서론에서 "이 글에서 다룰 내용은..." 나열하지 않는다. 바로 본론.
- 섹션 3~5개. 한 섹션이 스크롤 2번 넘기면 분리.
- 마지막에 짧은 요약. 3줄 이내.
- 코드 블록 바로 위에 한 줄로 맥락.

## 분량
- 1,500~3,000자 (한국어 기준). 수식/코드 제외.
- 한 문단 3~4줄 max.
- 넘기면 시리즈로 분리.

## 시각 자료
- Mermaid 다이어그램 적극 활용 (자동 렌더링 지원).
- 표는 비교/정리용.
- 수식이 3줄 이상 연속되면 중간에 해석 한 줄 끼우기.

## 시리즈별 템플릿

### Paper Deep Dive
```markdown
---
title: "[핵심을 담은 한국어 제목]"
date: YYYY-MM-DDT09:00:00+09:00
series: ["Paper Deep Dive"]
tags: [키워드들]
summary: "한 줄 요약"
author: "khrhee"
---

## 왜 이 논문을 읽었나

(동기. 2~3문장. 개인적 맥락 포함.)

## 핵심 아이디어

(논문의 main contribution. 수식 포함.)

## 방법론

(구체적 방법. 수식이 중심. 직관은 수식 해석으로.)

## 실험에서 눈에 띈 것

(전체 결과 나열 아님. 인상적이었던 포인트만.)

## 한계와 생각

(저자의 한계 + 내 비판. 솔직하게.)

## 정리

- 포인트 1
- 포인트 2
- 포인트 3
```

### Methods Explained
```markdown
---
title: "[방법론 이름]: [한 줄 설명]"
date: YYYY-MM-DDT09:00:00+09:00
series: ["Methods Explained"]
tags: [키워드들]
summary: "한 줄 요약"
author: "khrhee"
---

## 문제

(이 방법이 필요한 상황. 구체적 예시.)

## 수학적 정의

(formal definition. 수식이 이 섹션의 핵심.)

## 직관

(수식이 의미하는 것을 한 문단으로.)

## 구현

(코드. 공개 데이터 또는 시뮬레이션.)

## 실무에서 주의할 것

(전제 조건, 흔한 실수.)

## 정리

- 포인트 1
- 포인트 2
- 포인트 3
```

### Clinical ML Pitfalls
```markdown
---
title: "[함정 이름]"
date: YYYY-MM-DDT09:00:00+09:00
series: ["Clinical ML Pitfalls"]
tags: [키워드들]
summary: "한 줄 요약"
author: "khrhee"
---

## 상황

(문제를 만나는 전형적 상황.)

## 뭐가 문제인가

(수학적/통계적 설명.)

## 어떻게 잡는가

(탐지 방법. 코드 포함 권장.)

## 체크리스트

(실무에서 바로 쓸 수 있는 체크 항목.)

## 정리

- 포인트 1
- 포인트 2
- 포인트 3
```

## 보안 규칙

### 절대 금지
- 회사 데이터 수치 (환자 수, 코호트명, 성능 지표)
- 미발표 논문 핵심 결과
- 특허 출원 중인 아키텍처 세부사항
- 제품화 관련 구현 디테일

### 허용
- 공개 논문 인용 및 비판적 리뷰
- 공개 데이터셋 (MIMIC, PhysioNet 등) 예시
- 일반 방법론 설명 및 튜토리얼
- 간접 경험 공유 ("실무에서 이런 패턴을 자주 봤다" 수준)
- 시뮬레이션/합성 데이터 코드 예시

## 안티패턴 (이렇게 쓰면 안 된다)

- "이번 포스트에서는 A, B, C에 대해 알아보겠습니다" → 바로 A부터 시작
- "쉽게 말하면~", "간단히 정리하면~" → 수식으로 정의하고 해석
- "많은 연구에서~", "최근 연구에 따르면~" → 구체적 논문 인용
- 감정 없는 평서문 나열 → 중간에 생각 끼우기
- "결론적으로 이 방법은 매우 유용합니다" → 구체적으로 언제 쓰고 언제 안 쓰는지
