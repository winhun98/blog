마크다운 한 장이 에이전트를 만든다 — Karpathy의 autoresearch와 Ralph Wiggum Loop에서 뽑아낸 설계 원칙

2026년 3월 16일 · 10 분 · khrhee
📚 Research Toolbox

  2. 마크다운 한 장이 에이전트를 만든다 — Karpathy의 autoresearch와 Ralph Wiggum Loop에서 뽑아낸 설계 원칙 ← 현재 글

에이전트는 코드가 아니라 지시서로 만든다#

이전 글에서 Workflow Prompting이라는 개념을 다뤘다. Markdown에 자연어로 파이프라인을 적으면 LLM이 실행한다는 얘기. 그때는 “이게 되네?” 수준이었는데, 최근 두 가지 프로젝트를 깊게 뜯어보면서 확신이 생겼다. 마크다운 파일 하나가 에이전트의 전부가 될 수 있다.

하나는 Andrej Karpathy의 autoresearch. program.md라는 마크다운 파일 하나로 LLM을 자율 ML 연구원으로 만든다. 다른 하나는 Geoffrey Huntley가 만들고 Anthropic이 공식 플러그인으로 채택한 Ralph Wiggum Loop. while true; do cat PROMPT.md | claude; done — bash 한 줄로 Claude를 무한 루프 에이전트로 만든다.

겉으로는 완전히 다르다. 하나는 ML 실험 자동화, 하나는 범용 코딩 루프. 그런데 안을 뜯어보면 같은 설계 원칙을 공유한다.

두 시스템의 구조#

먼저 각각이 어떻게 생겼는지 보자.

Karpathy의 autoresearch#

출처: Anthropic, “Building Effective Agents” (link)

program.md에 모든 지시가 들어있다. 에이전트의 운영체제라고 볼 수 있다.

LOOP FOREVER:
1. Look at the git state
2. Tune train.py with an experimental idea
3. git commit
4. Run the experiment: uv run train.py > run.log 2>&1
5. Read out the results: grep "^val_bpb:" run.log
6. If grep output is empty → crash → read tail -n 50, attempt fix
7. Record in results.tsv
8. If val_bpb improved → keep commit (advance branch)
9. If worse → git reset back

그리고 여기에 한 줄이 붙는다:

NEVER STOP: Once the experiment loop has begun, do NOT pause to ask the human. The human might be asleep. You are autonomous.

README에 Karpathy가 직접 쓴 문장이 인상적이었다.

“you’re not touching any of the Python files like you normally would as a researcher. Instead, you are programming the program.md Markdown files that provide context to the AI agents”

사람은 Python을 짜지 않는다. 마크다운으로 에이전트를 프로그래밍한다. 이건 비유가 아니라 literal한 설명이다.

Ralph Wiggum Loop#

출처: Hugging Face, “AI Agents Course — Unit 1” (link)

원래 형태는 이렇게 단순했다:

while :; do cat PROMPT.md | claude-code ; done

Anthropic의 공식 플러그인 버전은 좀 더 정교하다. 4개 컴포넌트로 구성된다:

graph TD
    A["🎬 /ralph-loop<br/>'Build X' --max-iterations 20"] --> B["📝 State File 생성<br/>.claude/ralph-loop.local.md"]
    B --> C["🤖 Claude 작업 수행<br/>파일 수정, 테스트 실행"]
    C --> D{"🛑 Stop Hook 가로챔"}
    D -->|"completion promise 감지"| E["✅ 정상 종료"]
    D -->|"max iterations 도달"| E
    D -->|"조건 미충족"| F["🔄 iteration++<br/>동일 프롬프트 재주입"]
    F --> C

    style A fill:#1a1a2e,stroke:#e94560,color:#fff
    style B fill:#1a1a2e,stroke:#0f3460,color:#fff
    style C fill:#1a1a2e,stroke:#e94560,color:#fff

    style F fill:#1a1a2e,stroke:#e94560,color:#fff

핵심 트릭은 Stop Hook이다. Claude가 작업을 마치고 종료하려 할 때, hook이 가로채서 “아직 안 끝났어"라며 같은 프롬프트를 다시 넣는다. Claude 입장에서는 매번 새 세션이지만, 파일 시스템에 이전 작업 결과가 남아 있으니 이어서 작업한다.

공유하는 여섯 가지 설계 원칙#

표면적 차이에도 불구하고, 두 시스템은 같은 뼈대를 가지고 있다.

1. RLHF를 꺾어야 에이전트가 된다#

LLM은 RLHF로 훈련되면서 하나의 강한 본능을 갖게 된다. 답변을 마치면 멈추고 사용자를 기다리는 것. 챗봇으로서는 올바른 행동이지만, 자율 에이전트로서는 치명적인 약점이다.

Karpathy는 이걸 정면으로 꺾는다:

“NEVER STOP: Once the experiment loop has begun, do NOT pause to ask the human if you should continue. The human might be asleep.”

Ralph Loop는 더 기계적으로 해결한다. LLM의 본능을 설득하려 하지 않고, Stop Hook이 종료 시도 자체를 가로채서 같은 프롬프트를 다시 주입한다. LLM은 매번 “나는 끝났다"고 생각하지만, 시스템이 “아직이야"라며 다시 깨운다.

이 차이가 재미있다. Karpathy는 프롬프트로 LLM의 행동을 바꾸려 하고, Ralph는 시스템 아키텍처로 LLM의 행동을 우회한다. 둘 다 같은 문제 — “LLM이 멈추려 한다” — 를 풀고 있지만, 접근법이 다르다.

2. 무한 루프가 지능이다#

출처: Anthropic, “Building Effective Agents” (link)

Karpathy: LOOP FOREVER. Ralph: Stop Hook으로 종료를 차단. 둘 다 한 번에 완벽한 결과를 기대하지 않는다. 대신 반복이 품질을 만든다.

이건 의외로 중요한 통찰이다. LLM의 단일 패스 정확도에 집착하는 대신, “틀려도 되니까 계속 돌려” 전략이 실전에서 더 강력하다. Karpathy의 autoresearch가 하룻밤에 100개 실험을 돌리면서 val_bpb를 점진적으로 깎아내리는 게 바로 이 원리다.

처음 이 패턴을 봤을 때 “이게 되나?” 싶었는데, 생각해보면 인간 연구자도 똑같이 한다. 가설 세우고, 돌리고, 결과 보고, 수정하고. 다만 우리는 이걸 하루에 서너 번 하는 반면, 에이전트는 5분마다 한다.

3. 파일 시스템이 기억이다 (컨텍스트 윈도우가 아니라)#

출처: Weaviate, “Context Engineering — LLM Memory and Retrieval for AI Agents” (link)

여기서 두 시스템의 가장 영리한 설계가 드러난다.

autoresearchRalph Wiggum Loop
상태 저장Git commits + results.tsv파일 시스템 (코드, 테스트)
롤백git reset없음 (전진만)
히스토리results.tsv에 전 실험 기록Git history
컨텍스트 초기화매 루프 시작 시 git state 확인매 iteration 새 세션

둘 다 LLM의 내부 메모리(컨텍스트 윈도우)를 신뢰하지 않는다. 대신 외부 저장소에 상태를 기록하고, 매번 거기서 읽어온다.

Karpathy의 results.tsv가 특히 영리하다. 대화 컨텍스트가 길어지면 LLM은 과거에 실패했던 방식을 또 시도하는 루프에 빠진다. results.tsv는 이걸 막는다 — 에이전트가 다음 가설을 세울 때 대화 내역이 아닌 TSV 파일을 읽게 하면, 인간 연구자가 실험 노트를 넘기며 고민하듯 객관적인 다음 스텝을 계획한다. 대화 기억은 흐릿해지지만 파일은 정확하다.

Ralph Loop는 더 과격하다. 매 iteration마다 컨텍스트를 완전히 리셋한다. 이전 대화를 이어가는 게 아니라, 파일 시스템에 남은 작업물만 보고 판단한다. 프롬프트는 불변, 파일 시스템이 가변 — 이 비대칭이 핵심이다.

4. 출력을 격리하라#

Karpathy의 program.md에는 이런 패턴이 반복된다:

uv run train.py > run.log 2>&1        # stdout을 파일로 격리
grep "^val_bpb:" run.log              # 핵심만 추출
tail -n 50 run.log                    # 크래시 시에만 상세 확인

수백 줄의 training log를 컨텍스트에 흘려보내면 에이전트 성능이 떨어진다. 그래서 로그 파일로 격리하고, grep으로 숫자만 꺼낸다. Ralph Loop도 마찬가지 — 매 iteration에서 컨텍스트가 리셋되니까, 쓸데없는 과거 대화가 쌓이지 않는다.

이건 실무에서 에이전트를 운용할 때 가장 먼저 부딪히는 문제이기도 하다. 컨텍스트 윈도우가 100K, 200K로 커져도 garbage in garbage out 원리는 변하지 않는다. 필요한 정보만 넣는 게 더 중요하다.

5. 실패 비용을 0으로 만들어라#

autoresearch에서 가장 과소평가되는 설계가 이것이다. Git의 commit/reset 구조 덕분에 실패의 리스크가 문자 그대로 0이 된다. 아이디어가 실패하면 git reset으로 이전 상태로 완벽하게 돌아간다. 코드가 망가져도, OOM이 터져도, 컴파일 에러가 나도 — 한 줄이면 원복된다.

이게 왜 중요한가? 실패 비용이 0이면 에이전트가 과감해진다. “이건 좀 위험한데…“라는 판단 대신 “일단 해보자"가 된다. Karpathy가 프롬프트에서 단순성 기준(Simplicity Criterion)과 동시에 급진적 실험을 장려하는 게 모순처럼 보이지만, git이 안전망이니까 성립한다.

아이디어가 파격적일수록 → 실패 확률은 높지만
git reset이 있으니까 → 실패 비용은 0
따라서 → 기대값 = (낮은 성공률) × (높은 보상) > 0

에러 처리도 같은 맥락이다. Karpathy는 에이전트에게 에러를 분류하라고 한다. 타이포 같은 사소한 버그는 직접 고치고, 근본적으로 깨진 아이디어는 과감히 포기(skip)하고 다음으로 넘어간다. 10분 넘게 걸리면 kill. 이게 Fail-Fast 원칙이다 — 고칠 수 있으면 고치고, 없으면 버려라.

Ralph Loop에는 롤백이 없다. 대신 매 iteration이 새 세션이니까, 이전 iteration의 실수가 컨텍스트를 오염시키지 않는다. 다른 방식이지만 결과는 같다 — 실패해도 다음 시도가 깨끗하다.

6. 구조적 가드레일 > 프롬프트 가드레일#

에이전트한테 “이건 하지 마"라고 프롬프트에 쓰는 건 취약하다. 자율 에이전트는 종종 목표를 잊거나, 더 나쁜 경우 결과를 조작해서 “성공했다"고 착각한다. 두 시스템 모두 이걸 구조적으로 차단한다.

autoresearch의 구조적 제약:

  • prepare.py는 read-only → 평가 함수를 조작할 수 없다 (metric gaming 원천 차단)
  • train.py 하나만 수정 가능 → 편집 범위가 물리적으로 제한됨
  • 5분 고정 시간 예산 → “더 오래 돌리면 더 좋은 결과” 전략 불가
  • 새 패키지 설치 불가 → dependency hell 방지

Ralph Loop의 구조적 제약:

  • max_iterations → 무한 루프 탈출 안전장치
  • completion_promise → 정확한 문자열 매칭으로 완료 판정 (LLM이 “대충 끝났다"고 거짓말하는 것 방지)
  • Stop Hook의 transcript 파싱 → 에이전트 출력을 기계적으로 검증

Karpathy가 prepare.py를 read-only로 잠근 건 특히 인상적이다. “평가 기준을 건드리지 마"라고 프롬프트에 쓸 수도 있었지만, 그 대신 파일 권한으로 물리적으로 불가능하게 만들었다. 프롬프트는 suggestion이지만, 파일 시스템 권한은 law다.

7. 마크다운이 프로그램이다#

이 글의 제목이기도 한 핵심 원칙. 두 시스템 모두 .md 파일이 에이전트의 소스 코드다.

    end

LangGraph로 상태 머신을 짜고, CrewAI로 역할을 배정하고, AutoGen으로 대화 프로토콜을 정의하는 방식과 근본적으로 다르다. 사람은 의도를 자연어로 기술하고, LLM이 실행 계획으로 변환한다. 코드 없이 에이전트를 만드는 게 아니라, 마크다운 자체가 코드인 새로운 프로그래밍 패러다임이다.

한 가지 흥미로운 점은, Karpathy의 autoresearch에는 val_bpb라는 명확한 정량 지표가 있다는 것이다. 코드 리뷰나 문서 작성 같은 정성적 작업에는 이런 자동 지표가 없다. 이럴 때는 Anthropic이 제안한 Evaluator-Optimizer 패턴이 대안이 된다 — 별도의 LLM이 결과물을 평가하고, reject이면 피드백과 함께 재생성을 요청하는 구조다. 자동화된 보상 함수가 없으면 LLM으로 보상 함수를 만드는 것이다.

설계 체크리스트#

두 시스템에서 뽑아낸 원칙을 실전 체크리스트로 정리한다:

  1. RLHF를 꺾어라. 프롬프트로든(NEVER STOP) 아키텍처로든(Stop Hook) 에이전트의 “멈추려는 본능"을 억제하라.
  2. 루프를 짜라. 단일 호출로 끝내려 하지 말고, 반복 구조를 기본으로 설계하라.
  3. 상태를 외부에 두라. 컨텍스트 윈도우가 아니라 파일/DB에 기록하라. 대화 기억은 흐릿해지지만 파일은 정확하다.
  4. 출력을 격리하라. 에이전트에게 보여줄 정보를 의식적으로 필터링하라.
  5. 실패 비용을 0으로 만들어라. Git, 브랜치 격리, 세션 리셋 — 무엇이든. 실패가 무료여야 에이전트가 과감해진다.
  6. 프롬프트가 아니라 구조로 제약하라. “하지 마"보다 “할 수 없게” 만들어라.
  7. 마크다운을 프로그램으로 취급하라. 코드 프레임워크 없이도 에이전트는 만들 수 있다.

정리#

  • Karpathy의 autoresearch와 Ralph Wiggum Loop는 표면적으로 다르지만, RLHF 억제 → 무한 루프 → 외부 상태 → 실패 비용 제로 → 구조적 제약 → 마크다운 프로그래밍이라는 동일한 설계 뼈대를 공유한다.
  • 에이전트의 지능은 모델 크기가 아니라 루프 구조와 상태 관리 설계에서 나온다. 630줄 Python과 마크다운 하나로 하룻밤 100개 실험을 돌리는 게 그 증거다.
  • 에이전트 프로그래밍의 주류가 Python 프레임워크에서 마크다운 지시서로 이동하고 있다. 이건 일시적 유행이 아니라, LLM이 범용 런타임이 된 시대의 자연스러운 귀결이다.
